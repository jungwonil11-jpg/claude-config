# 연습 — 맨땅 헤딩 코딩 연습 사이클

두 가지 모드로 동작한다:
- `/연습` : 빈칸 골격 생성
- `/연습 채점` : 사용자가 작성한 코드 채점

빈칸 형식은 **항상 fill-in-the-blank (`_____`)**. 퍼센티지로 빈칸 양만 조절.

---

## 빈칸 비율과 자리 정의

**핵심 원칙**: 퍼센티지는 **빈칸 양**의 비율임. 카테고리 누적 ❌, 토큰 풀 × 비율 ✅.

### 1. 빈칸 가능 토큰 풀

선택 기능 메소드 안의 모든 토큰 중, 다음만 **빈칸 X (절대 안 비움)**:

- **구조 키워드**: `try`, `catch`, `if`, `else`, `for`, `while`, `do`
- **구두점/연산자**: `{`, `}`, `(`, `)`, `;`, `=`, `,`, `.`, `:`
- **패키지/import 구문** 전체
- **클래스 선언 + 클래스 레벨 어노테이션** (`@RestController`, `@RequestMapping("/...")` 등 — 선택 기능 메소드 범위 밖)
- **필드 선언부** (`@Autowired private GuestBookService guestBookService;`)

그 외 모든 토큰 = **빈칸 가능 자리** (= "빈칸 풀").

빈칸 풀에 들어가는 자리 예시:
- 메소드 레벨 어노테이션 + 파라미터: `@GetMapping`, `"/list"`
- 메소드 시그니처 리턴 타입: `DataVO`
- 변수 선언 타입: `DataVO`, `List<GuestBookVO>`, `Exception`
- 생성자 호출: `new`, 클래스명
- 메소드 호출: 호출명 자체
- getter/setter
- 메소드 호출 인자 (값, 문자열, Boolean)
- 조건 연산자: `==`, `||`, `&&`
- 값 토큰: `null`, `Boolean.TRUE/FALSE`, 숫자, 문자열
- 흐름 키워드: `return`, `throw`, `break`, `continue`

### 2. 비율 적용 방식

선택 기능 메소드의 빈칸 풀 토큰 수가 N개라면:

| 비율 | 빈칸 개수 | 빈칸 X 추가 자리 |
|---|---|---|
| **33%** | ≈ N × 0.33 | 변수명, 컨트롤러 메소드명, `public` 접근제어자, 시그니처 리턴타입 일부 살림 |
| **67%** | ≈ N × 0.67 | 변수명, 컨트롤러 메소드명, `public` 접근제어자 살림 |
| **100%** | N개 전부 | 위 자리들도 다 빈칸 |

같은 토큰 여러 번 등장해도 100%에선 전부 비움. **"같은 토큰 첫 등장만 빈칸" 룰은 폐기됨.**

### 3. 33% / 67% 선별 우선순위 (P1 → P8)

비율 채울 때 위에서부터 비움:

1. **Service/Mapper 메소드 호출명** (5-파일 묶음 핵심)
2. **getter/setter 첫 등장**
3. **표준 라이브러리 메소드 호출** (`isEmpty`, `getMessage`, `equals` 등)
4. **어노테이션 파라미터** (`"/list"`, `MembersVO.class` 등)
5. **같은 토큰 중복 등장** (setter 2번째 이후 등)
6. **문자열 리터럴** (사용자 메시지)
7. **변수 타입, 생성자(`new`), 시그니처 리턴 타입**
8. **조건 연산자(`==`/`||`/`&&`), 값 토큰(`null`/`Boolean.*`/숫자), 흐름 키워드(`return`/`throw`)**

100%는 P1~P8 전부 + 변수명/컨트롤러 메소드명/`public`까지.

### 4. 카테고리 ① — 메소드명 빈칸 범위 (5-파일 묶음 핵심)

- ✅ 메소드 호출 위치 (예: `guestBookService.guestBookList()`)
- ✅ 서비스/매퍼 **인터페이스** 시그니처의 메소드명
- ✅ `@Override` 메소드 시그니처의 메소드명 (ServiceImpl)
- ✅ XML `<select id="...">` 등 매퍼 쿼리 id 속성값
- ❌ **컨트롤러 클래스의 메소드명**은 33%/67%에선 빈칸 X (자유 작명이라 채점 모호). 100%에서만 빈칸.

> 의도: Controller가 부른 메소드명이 ↔ Service 시그니처 ↔ 임플 `@Override` ↔ Mapper 시그니처 ↔ XML `id` 가 **전부 정확히 같아야 동작**한다는 일관성을 5번 채우면서 자연스럽게 익힘.

### 2-A. 5-파일 묶음 모드

#### a. PROGRESS 확인

`practice/PROGRESS.md` 읽음. 없으면 자동 생성 (구조는 아래 "PROGRESS.md 구조" 참고). 기능 목록은 `src/` 컨트롤러를 분석해서 동적으로 채움.

#### b. 기능 + 퍼센티지 선택

해당 컨트롤러 도메인 매트릭스 출력. UI 선택지 X.

```
컨트롤러 선택. 5-파일 묶음 모드.

[Members] 진척도:
| # | 기능 | 33% | 67% | 100% |
|---|---|---|---|---|
| 1 | 로그인 | ⬜ | ⬜ | ⬜ |
| 2 | 토큰 재발급 | ⬜ | ⬜ | ⬜ |
```

- 잠금(🔒) 기능은 선택 못 함. 강사 코드에 해당 메소드가 없으면 자동 잠금.
- 이미 ✅인 셀도 다시 풀 수 있음 (반복 학습 허용).
- 잠금/해제는 PROGRESS.md를 사용자가 직접 수정해서 변경 가능.

#### c. 빈칸 생성

회차 폴더 생성. 명명 규칙:
```
practice/YYYY-MM-DD_<도메인>-<기능slug>_<%>/
```
- `도메인`: 컨트롤러 도메인 (예: `guestbook`, `members`)
- `기능slug`: 기능 영문명 케밥케이스 (예: `list`, `insert`, `login`, `refresh`, `mypage`, `logout`, `detail`)
- `%`: `33`, `67`, `100`
- 예: `practice/2026-05-15_guestbook-list_33/`, `practice/2026-05-15_members-login_67/`

**같은 회차 폴더 이미 있으면 `_2`, `_3` 자동 추가** (반복 학습용. 답안 보존).

5개 파일 모두 원본 확장자로 생성.

| 파일 | 처리 방식 |
|---|---|
| 컨트롤러 | 선택 기능의 메소드만 빈칸. 다른 기능 메소드는 `src/` 실제 코드 그대로 박음. |
| 서비스 인터페이스 | 모든 기능 시그니처 그대로 박음 (빈칸 없음). |
| 서비스 임플 | 선택 기능 메소드만 빈칸. 다른 기능 구현은 그대로 박음. |
| 매퍼 인터페이스 | 서비스 인터페이스와 동일. |
| 매퍼 XML | 선택 기능 SQL만 빈칸. 다른 쿼리는 그대로. |

빈칸은 위 "빈칸 비율과 자리 정의" 룰에 따라 선택 퍼센티지에 맞춰 표시.

**잠긴 기능은 메소드/쿼리 자체를 안 넣음** (나중에 추가될 자리 비어둠).

#### d. README.md 생성

회차 정보:
- 선택 기능 + 퍼센티지
- 빈칸 카테고리 매핑 (어떤 카테고리가 빈칸인지)
- "IDE 열고 `_____` 채우기. 끝나면 `/연습 채점`."

#### e. PROGRESS.md 갱신

해당 셀을 ⌛(진행 중)로 변경. `practice/PROGRESS.md`의 `마지막 갱신` 날짜도 오늘로.

**이전에 다른 % 셀이 ⌛인 경우 그대로 유지** — 각 회차 폴더가 독립적으로 살아있으니 사용자가 언제든 돌아가서 풀 수 있음. 즉 같은 기능 행에 `⌛ | ⌛ | ⬜` 같이 동시 진행 가능.

## 모드 B: `/연습 채점` — 채점

### 1. 연습 파일 찾기

`practice/` 안에서 **가장 최근 수정된(mtime) 회차 폴더**를 찾음. 안의 파일들 읽음 (`README.md`, `SCORE_*.md` 제외).

- 회차 정보(기능/%)는 폴더명에서 자동 파싱:
  - `2026-05-15_guestbook-list_67/` → 5-파일 묶음, 도메인: GuestBook, 기능: 방명록 리스트, 67%
  - `2026-05-15_jwt-util_33/` → 단일 파일, JwtUtil, 33%
- 사용자가 특정 회차를 채점하고 싶으면 폴더명을 직접 지정 가능 (예: `/연습 채점 2026-05-15_guestbook-list_33`).

### 2. 실제 코드와 비교
파일명 매칭으로 `src/**/{파일명}` 찾아서 비교.

### 3. 채점 기준

빈칸 위치(`_____`였던 자리)별 O/X:

| 자리 유형 | 채점 기준 |
|---|---|
| **5-파일 묶음 메소드명** (Service/Mapper 인터페이스 + ServiceImpl @Override + 호출 위치 + XML `id`) | **사용자가 5곳에서 일관되게 사용했으면 O**. 강사 코드와 다른 이름이어도 동작하면 인정 (의도가 "5곳 일치"이지 "강사와 일치"가 아님) |
| getter/setter, 필드 접근 | 완전 일치 (Lombok 자동 생성이라 변경 불가) |
| 어노테이션 이름·파라미터 | 완전 일치 (Spring이 인식하는 이름이라 변경 불가) |
| SQL 식별자 (테이블·컬럼) | 완전 일치 (대소문자 무시) — 실제 DB 스키마와 일치해야 동작 |
| 문자열 리터럴 | 의미 비슷하면 O |
| 변수 타입, 시그니처 리턴 타입, 생성자 | 완전 일치 (실제 클래스명이라 변경 불가) |
| 조건 연산자, 값 토큰, 흐름 키워드 | 의미 동일하면 O (예: `Boolean.FALSE` ↔ `false` 허용) |
| 변수명, 컨트롤러 메소드명 (100%만) | 사용자가 일관되게 사용했으면 O |

**원칙**: "동작 가능하면 O". 자바/Spring/MyBatis 입장에서 컴파일·런타임에 문제없으면 정답으로 인정. 강사 코드와의 비교는 **스타일 차이일 뿐 오답 아님**.

점수: `맞은 빈칸 / 전체 빈칸`

### 4. 채점 결과 저장 — 학습형 SCORE.md

채점은 단순 O/X 나열이 아닌 **오답마다 정답 + 왜 + 외우는 법**까지 박힌 학습 노트로 작성. 사용자가 SCORE.md만 읽어도 그 빈칸을 다시 안 헷갈리게 만드는 게 목적.

#### SCORE.md 구조

**파일 단위로 묶음**. 개념 묶음·잘한 점·다음에 할 것 섹션 **금지**.

**파일명 규칙**: `SCORE_<N>.md` (N=1부터 시작, 같은 회차 폴더에서 재채점할 때마다 +1).
- 첫 채점 → `SCORE_1.md`
- 두 번째 채점 → `SCORE_2.md` (이전 `SCORE_1.md`는 보존 — 진척 비교용)
- 채점 시 **해당 회차 폴더 안의** `SCORE_*.md` 개수만 세서 다음 번호 결정. 다른 폴더의 SCORE 개수는 무관 — 카운터는 **회차 폴더별로 독립**
- 예: list_67 폴더에 SCORE_2까지 있어도 insert_67 첫 채점은 `SCORE_1.md`

```markdown
# 채점 결과 #<N> — YYYY-MM-DD <도메인> <기능> <%>

(이전 채점이 있으면 1줄: `> 이전 채점: SCORE_<N-1>.md (X/Y, Z%)`)

## 점수
**X / Y (Z%)**

| 파일 | 점수 |
| ... | ... |

---

## <파일명1> (X/Y)

### L<라인>: `<오답>` → `<정답>`
- **왜 틀렸나**: 컴파일러·런타임 관점에서 근본 원인 (1~3줄)
- **외우는 법**: 다음번에 안 틀리게 하는 패턴/연상법 (1~2줄)

### L<라인>: `<오답>` → `<정답>`
(반복)

---

## <파일명2> (X/Y)
(반복)
```

#### 작성 룰

- **파일 단위 묶음 고정**: Controller → Service → ServiceImpl → Mapper → XML 순. 5-파일 묶음이면 5개 섹션, 단일 파일이면 1개 섹션
- **각 오답마다 정답 박을 것**. 빈칸 자리의 정답 토큰만 (메소드 전체 코드 통째로 X — 다음 회차 풀 때 보존)
- **맞은 빈칸은 SCORE.md에 안 적음**. 틀린 것만 나열
- **잘한 점 / 다음에 할 것 / 개념별 묶음 섹션 박지 말 것**
- **왜 틀렸나·외우는 법은 매 오답마다 1~3줄로 짧게**. 똑같은 이유로 여러 군데 틀렸으면 첫 번째에만 자세히 쓰고 다음부터는 "같은 이유" 한 줄로 처리
- 미작성(`_____` 그대로) 빈칸도 오답으로 처리. 정답 박고 "미작성" 이유 한 줄

#### 채팅 응답

채팅엔 다음만:
1. 총점 (한 줄)
2. `[SCORE.md](path) 참고` 안내

오답 나열·요약·시급 개념 등 **채팅에 박지 말 것**. 사용자가 SCORE.md를 직접 읽음.

### 5. PROGRESS.md 갱신 (5-파일 묶음일 경우)

`practice/YYYY-MM-DD/README.md`에서 회차 정보 읽어서 매칭:
- 만점 → ✅
- 부분 점수 → ⌛ 유지
- 0점 → ⬜ 회귀

단일 파일 모드는 PROGRESS.md 안 건드림.

---

## PROGRESS.md 구조

위치: `practice/PROGRESS.md` (날짜 폴더 밖, 마스터 진척도)

PROGRESS.md가 없으면 `/연습` 첫 실행 시 자동 생성. 기능 목록은 `src/`의 컨트롤러를 분석해서 동적으로 채움.

```markdown
# 5-파일 묶음 연습 진척도

마지막 갱신: YYYY-MM-DD

## GuestBook (`/guestbook`)
| # | 기능 | 33% | 67% | 100% |
|---|---|---|---|---|
| 1 | 방명록 리스트 | ⬜ | ⬜ | ⬜ |
| 2 | 방명록 등록 | ⬜ | ⬜ | ⬜ |
| 🔒 | 방명록 상세 (강사 미구현) | | | |

## Members (`/members`)
| # | 기능 | 33% | 67% | 100% |
|---|---|---|---|---|
| 1 | 로그인 | ⬜ | ⬜ | ⬜ |
| 2 | 토큰 재발급 | ⬜ | ⬜ | ⬜ |
| 3 | 마이페이지 | ⬜ | ⬜ | ⬜ |
| 🔒 | 로그아웃 (강사 미구현) | | | |

## 표기
- ⬜ 미진행
- ⌛ 진행 중 (빈칸 생성 후 채점 전)
- ✅ 만점 완료
- 🔒 잠김 (강사 미구현)

## 잠금 해제 방법
강사가 새 기능을 강의/구현한 후, 위 표에서 해당 행의 🔒를 #(다음 번호)로 바꾸고 셀을 ⬜로 추가.
```

### 학습 곡선 기준 기본 순서 (PROGRESS 자동 생성 시 적용)

신규 PROGRESS.md 생성 시 아래 순서로 행 배치. 학습 곡선 + 의존성 고려.

**GuestBook** (JWT 무관 → 진입용):
1. 리스트 조회 (단순 SELECT)
2. 등록 (INSERT + `@RequestBody`)
3. 상세 — 강사 미구현이면 🔒

**Members** (JWT 필요):
1. 로그인 (BCrypt + JwtUtil + 2테이블)
2. 토큰 재발급 (로그인 패턴 재활용)
3. 마이페이지 (SecurityContext)
4. 로그아웃 — 강사 미구현이면 🔒
